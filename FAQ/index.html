    集群总览¶

    惠普管理节点：CAELNSE1\2 （用的惠普CMU进行系统分发，克隆）

    计算节点：CAEOPE01-82

    GPFS server：CAESNS01\02

    PBS pro + PBS WORKS：CAELNSE1

    前后处理节点（显示节点）：CAEGNS01、02、03、04、05、06、07

    用户管理NIS server：CAELNSE1

    其他说明：所有节点配置了nfs + autofs（管理节点会挂载所有节点，计算节点会根据作业调度到哪几台，在哪几台进行相应节点的挂载） /etc/exports /etc/auto.master /etc/auto.pbs

    一、显示节点打不开桌面应用¶

    现象：

    打开图形，无法进入图形桌面，并自动加载应用

    排查：

    1.通过页面Altair Compute Manager登录pbsadmin/1qaz2wsx，提交一个图形作业

    2.进入监控，选中提交的作业，点击图形，查看下方的计算节点

    3.命令行su - pbsadmin，并ssh到上面的计算节点，本次案例中会出现此过程非常缓慢，考虑是gpfs并行文件系统存在异常。

    4.利用命令mmfsadm dump waiter 检查gpfs文件系统延迟情况，以下是截图详解

    5.联系用户尝试重启延迟的节点，重启后检查。本案例最终排查到gpfs server02节点延迟很大，重启后还是延迟，但桌面可以打开了，让客户联系存储硬件工程师检查一下存储链路是否正常。

    二、作业目录报stale file handle错误¶

    报错截图：

    问题分析：

    从报错截图可以看到作业到共享目录出现错误，出错的节点分别为caeope56、caeope57、caeope60。说明当时这三个节点gpfs挂载出现问题，此时可先ssh到三个节点上df -h看下存储还在不在，再到CAESNS01节点上查看gpfs日志，最后到各问题节点上看gpfs日志，综合双方日志进行分析。

    问题解决：

    先说明一下，gpfs日志主要关注俩个，一个是gpfs重启前的日志，即/var/adm/ras/mmfs.log.previous，一个是gpfs重启后的日志，即/var/adm/ras/mmfs.log.latest。这里可以先到CAESNS01节点上查看gpfs日志，用命令cat /var/adm/ras/mmfs.log.latest |grep -i caeope60，关注问题出现时间点和内容，

    再ssh到问题节点，通过gpfs日志查到gpfs重启的时间点，然后根据时间点再去查Linux系统在那个时间点是不是有问题，Linux系统日志在/var/log/messages。本案例最终查到是用户提交的作业模型过大造成内存溢出导致gpfs重启。

    三、作业处于一直排队或wait状态如何处理？¶

    1、如果作业是排队状态，则可能是正常的也可能是不正常的；如果作业处于wait状态，则一定是不正常的。

    2、PBS和lsf一样，也分为server端和client端，奇瑞pbs的server端在 CAELNSE1 节点，作业通过web页面提交，如图：

    作业提交到完成流程：作业被用户提交后→server端根据提交的信息分配计算节点→server端scp source文件到计算节点（包括license信息等）→计算节点开始计算→计算完成后复制到本地缓存（多个节点之间通过autonfs共享缓存目录）→最后将计算结果scp到gpfs共享目录。

    3、一直wait状态原因：大多数是由于“server端scp source文件到计算节点（包括license信息等）”这一步出错，通过在server节点CAELNSE1查看日志 /var/spool/PBS/server_logs（server日志看server_logs，计算日志看mom_logs）/20201118（日期） 判断，

    scp失败可能原因：

    ①节点没做互信（用用户账号ssh试试，还有就是用户第一次登陆提示yes/no也会影响）

    ②被拷贝的计算节点down机

    ③gpfs或autofs故障

    ④还有可能是用户提交的作业数据量太大，本地/scratch磁盘空间不足，可以df -h查看节点/目录使用率，此时建议用户把作业提交到本地磁盘更大的节点计算或者指定scratch 目录为/gs/scratch目录。之后再时删除/scratch/目录中作业失败残留的目录。

    4、一直排队q状态原因：

    1）先根据作业号检查原因

    qstat -fx 作业id #根据id查看作业详细信息

    pestat|grep -i 节点名 #查看某个节点的作业信息（id、用户、platform、状态等）

    2）如果显示关注comment状态，此种情况是节点资源不够导致，让用户等待即可。

    3）如果不是以上原因导致，则可能是由于“server端scp source文件到计算节点（包括license信息等）”这一步出错，通过在server节点CAELNSE1查看日志 /var/spool/PBS/server_logs（server日志看server_logs，计算日志看mom_logs）/20201118（日期） 判断，

    scp失败可能原因：

    ①节点没做互信

    ②被拷贝的计算节点down机

    ③gpfs或autofs故障

    四、集成软件（starccm、VAOne）配置文件修改（license、mpi运行环境等）说明。¶

    因为pbs中软件都是集成到web界面上，web界面是由/var/spool/pas目录下的配置文件控制，所以本案例中VAOne软件相关配置文件也在此目录下。

    cd /var/spool/pas/repository/applications

    cd VAOne/runtime

    vim start.sh

    license相关：

    export CDLMD_LICENSE_FILE=/gs/apps/CCM1406/ccm1406R8license.dat

    mpi相关：

    设置为走ib网络¶

    export MPIRUN_OPTIONS="-v -IBV -cpu_bind -e MPI_IB_PKEY=0x8df1,0x7fff"

    设置为走普通网络¶

    export MPIRUN_OPTIONS="-v -TCP -cpu_bind"

    之后进入cd /var/spool/pas/repository

    rm -rf time_stamp.txt

    五、资源池恢复和删除¶

    示例：49节点CPU更换后，需要调整下资源池，正常投入使用？

    恢复节点资源池：

    1.查看节点当前资源池名称。（此步可选）

    pestat|grep -i caeope49

    2.查看节点资源池和其他信息。（此步可选）

    qmgr -c "p n CAEOPE49"

    qmgr -c "p n CAEOPE50"

    3.调整节点资源池

    qmgr -c "set node CAEOPE49 resources_available.platform = E52667-16c-256G"

    4.把软件集成到资源池(资源池中的所有节点都要一个一个添加)

    qmgr -c "set node CAEOPE49 resources_available.pas_applications_enabled += Dyna-mmp"

    qmgr -c "set node CAEOPE50 resources_available.pas_applications_enabled += Dyna-mmp"

    ...

    5.把软件资源池添加到xml模板

    cd /var/spool/pas/repository/applications/

    cd Dyna-mmp/

    vim app-inp-Dyna-mmp.xml

    6.检查软件提交模板配置

    vim Dyna-mpp/submittime/presubmit.py

    7.检查软件是否加入到资源池其他配置文件中

    vim /var/spool/PBS/server_priv/resourcedef

    nastran type=long 检查是否有此行，没有则按此格式添加：软件名 type=long

    vim /var/spool/PBS/sched_priv/sched_config

    检查这里是否有新软件名，没有则添加

    7.查看资源池调整后信息

    qmgr -c "p n CAEOPE49"

    删除资源池：

    1.进配置文件目录

    cd /var/spool/pas/repository/applications/

    cd Abaqus-smp/

    2.编辑xml文件，注释或删除

    vim app-inp-Abaqus-smp.xml

    3.删除软件资源池(资源池中的所有节点都要一个一个删除)

    qmgr -c "set node CAEOPE49 resources_available.pas_applications_enabled -= Dyna"

    qmgr -c "set node CAEOPE50 resources_available.pas_applications_enabled -= Dyna"

    ...

    4.最后移动文件到tmp目录或删除time文件

    cd /var/spool/pas/repository

    mv time_stamp.txt /tmp/

    六、server节点df -h卡住¶

    大多数情况是某个节点无法ssh，nfs挂载出问题了，强行卸载问题节点的nfs即可。

    umount -l /pbs/CAEOPE25

    PS：那么如何知道是哪个节点挂了呢？俩个方法可以找到：

    1.通过命令pestat|grep -i down查找down机的节点

    2.第一种如果没找到，就用第二种查找，命令为

    shell strace df -h

    九、替换altair的license¶

    altair的license格式一般为：ALCH_CHAU_20210210_526858.dat

    替换步骤：

    #1、进入license目录，并将license传进来
cd /usr/local/altair/licensing13.2/
#2、给license执行权限
chmod +x ALCH_CHAU_20210210_526858.dat
#3、编辑license所在路径的配置文件
vim ltair-serv.cfg


    #4、重启altair服务
/etc/init.d/altairlmxd restart
#5、重启pbs服务
/etc/init.d/pbs restart

    第五步这里可能出现报错如下：

    这时候需要修改pbs执行进程为对应版本后再重启

    which qstat cd /opt/pbs/bin/ cp qstat.18 qstat

    /etc/init.d/pbs restart

    之后再修改回来重启pbsworks

    cp qstat.13 qstat

    #6、重启pbsworks
/etc/init.d/pbsworks restart
#7、查看license信息
/usr/local/altair/licensing13.2/bin/almutil -licstat -host mgt -port 6200

    如图能看到license到期时间则说明替换成功

    故障问题说明：

    十五、作业计算慢排查¶

    1.首先qstat -xf 作业号 查看comment=...，看看作业跑在哪几个节点。

    2.登录这几个节点查看cpu负载，负载高低根据cpu核数判断，比如cpu核数40，那么负载在40是正常的，超过40达到俩倍，这种就是pbs调度出了问题，pbs调度的时候没有判断好此节点正在跑的作业或残留的作业，继续把新作业调度上来，这样肯定就慢了。此时到CAELNSE1上用pestat|grep 作业跑的节点，查看此作业运行的用户，再到作业跑的节点上top并killall -u 掉其他用户的作业。

    小技巧：top命令显示用户名不全的时候会显示为xiaying+，top时按f进入显示设置，空格勾选uid选项，按q退出即可显示用户的uid，之后再id uid命令显示用户名即可。

    3.（可选）如果以上没有查到原因，可进一步排查，下线节点，对节点进行单节点的linpack测试和iozone测试。

    vim /var/spool/PBS/sched_priv/sched_conf

    加一行 job_sort_key: "higher HIGH" ALL

    # 之后按顺序执行
1.qmgr -c "set server scheduling =false"
2. cat /var/spool/pbs/sched_logs/20200429 查看最后一行是Leaving Scheduling Cycle
3. /etc/init.d/pbs status 获取pbs_sched is pid 82896
4. kill -9 82896
5. /etc/init.d/pbs status 确认pbs_sched not running
6. /etc/init.d/pbs start; /etc/init.d/pbs status 检查pbs_sched正常运行
7. qmgr -c "set server scheduling = true" 开启调度
